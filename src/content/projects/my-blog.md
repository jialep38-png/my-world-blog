## 项目定位

本项目是一个长期维护的个人内容站点，目标不是“搭一个能跑的页面”，而是建立一套可持续更新的内容系统：

- 支持长文（随笔）、短内容（絮语）、项目记录、小记四类内容
- 保持极简视觉，降低维护复杂度
- 通过静态部署降低运维成本
- 支持 RSS、评论、SEO 等长期运营所需能力

简而言之：这是一个可写、可看、可维护、可迭代的小圈系统。

## 背景与约束

### 背景

初始代码基于 `astro-whono` 主题，具备良好的基础结构，但包含较多模板痕迹（示例文案、示例图片、默认文档）。

目标不是“微调模板”，而是将其完整迁移为个人站点语义与内容。

### 约束

1. **最小侵入改造**：尽量复用主题已有架构，避免重写核心布局。
2. **内容先行**：优先完成内容模型和栏目体系，再补充视觉细节。
3. **发布导向**：每次关键改动后直接推送上线，确保线上即文档。
4. **低运维**：静态站点 + Git 驱动发布，不引入额外后端。

## 信息架构（IA）

当前站点按“内容类型”而非“时间线”组织，兼顾浏览效率与维护成本：

| 栏目 | 路由 | 目标读者行为 | 内容粒度 |
|---|---|---|---|
| 随笔 | `/essay/` | 深度阅读 | 长文 |
| 项目 | `/projects/` | 快速了解做过什么 | 中长文/结构化说明 |
| 絮语 | `/bits/` | 快速浏览动态 | 短文+图片 |
| 小记 | `/memo/` | 时间线式回看 | 短段落 |
| 归档 | `/archive/` | 按时间检索 | 索引 |
| 关于 | `/about/` | 建立身份与联系入口 | 固定页 |

这套 IA 的核心价值在于：**写作场景和阅读场景分离**。写作者可按内容类型投放，读者可按消费方式选择入口。

## 实施计划（Plan）

项目实际执行采用阶段化推进，避免一次性大改导致不可控回归。

### Phase 1：站点身份迁移

- 更新站点标题、作者、描述
- 更换头像资源及配置路径
- 统一“博客 / 小站”等命名为“小圈”语义

**产出**：站点主身份一致，导航和页面文案不再出现模板默认语义。

### Phase 2：模板痕迹清理

- 删除 demo 图片（bits/archive/memo/preview）
- 删除模板说明文档（英文 README、变更日志）
- 替换首篇示例内容

**产出**：仓库资源层去模板化，公共资源目录仅保留项目需要的文件。

### Phase 3：内容体系落地

- 创建并完善第一篇随笔
- 创建项目栏目首篇项目文档
- 调整 about 与侧栏引导语

**产出**：站点从“空壳模板”转为“可持续更新内容系统”。

### Phase 4：发布与回归

- 每次变更后提交并推送 `main`
- 触发托管平台自动部署
- 线上验收导航、内容页、资源引用

**产出**：建立“编辑即上线”的稳定发布节奏。

## 技术选型与取舍

### 为什么保留 Astro + 主题二次开发

- Astro 对内容型站点非常友好，Markdown 与内容集合能力成熟。
- 主题已有双栏布局、暗色模式、阅读模式、RSS 等可直接复用。
- 相较于从零开发，二次开发更快形成可发布版本。

### 为什么不引入数据库与后端

- 当前内容量与交互需求不需要动态存储。
- 静态部署更稳定，成本几乎为零。
- 对个人站点而言，复杂度应服务内容，而不是技术炫技。

### 评论系统选型

采用 Giscus（GitHub Discussions）而非自建评论后端：

- 不维护用户系统
- 可借助 GitHub 账号体系
- 数据可追踪、可迁移

## 目录与内容模型

### 内容目录

- `src/content/essay/`：随笔
- `src/content/projects/`：项目记录
- `src/content/bits/`：絮语
- `src/content/memo/`：小记

### 项目内容 Schema（摘要）

项目文档采用结构化 frontmatter，关键字段包括：

- `title` / `description` / `date`
- `status`（planned/in-progress/shipped/...）
- `stack`（技术栈）
- `links.github` / `links.demo`
- `tags` / `featured` / `slug`

这样做的价值是：后续可直接基于字段做筛选、排序、推荐，而不是依赖正文文本解析。

## 质量与可读性策略

为了让“项目记录”真正可读，而非流水账，本文档采用以下规范：

1. **先结论后细节**：先说做了什么，再说怎么做。
2. **结构化分段**：背景、约束、计划、实施、交付分层清晰。
3. **表格优先**：对比与清单类信息优先使用表格。
4. **术语统一**：全站统一使用“小圈”语义，不混用“博客/小站”。
5. **可复用表达**：避免只描述一次性动作，尽量抽象为可迁移方法。

## 风险与后续计划

### 已识别风险

| 风险 | 影响 | 处理策略 |
|---|---|---|
| 内容增长后导航拥挤 | 阅读效率下降 | 后续增加标签页或筛选入口 |
| 图片资源无统一规范 | 页面体积波动 | 统一压缩流程，优先 WebP |
| 单人维护节奏不稳定 | 更新中断 | 建立轻量周更机制（随笔/絮语） |

### 下一阶段计划

- 增加 2~3 篇项目文档，形成项目页的密度
- 为项目页增加“关联随笔”内容链接
- 继续完善 RSS 与 SEO 细节（canonical、sitemap 一致性）
- 按需补充 favicon 与品牌视觉资产

## 可迁移经验（给后续项目）

1. 主题改造优先做“语义替换”，再做“视觉替换”。
2. 先清理 demo 资源，再补真实内容，避免视觉噪音干扰判断。
3. 用结构化 frontmatter 为后续功能预留能力。
4. 小步上线比大改离线更稳：每次改动都能快速验证。
5. 个人站点的核心竞争力是持续输出，不是技术堆料。
