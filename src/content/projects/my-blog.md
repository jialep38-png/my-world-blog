## 摘要（先看这里）

本项目完成了从通用主题到个人站点的**全链路迁移**：

1. 完成站点身份接管（站点名、作者、头像、文案语义）
2. 清理模板痕迹（示例资源、默认文档、占位内容）
3. 建立可持续内容体系（随笔 / 项目 / 絮语 / 小记）
4. 接入发布闭环（Git 提交 → 主分支推送 → 自动部署）
5. 形成后续可扩展的内容模型（结构化 frontmatter + 项目字段）

这不是“美化模板”，而是一次针对内容站点的工程化落地。

---

## 一、项目目标与边界

### 1.1 目标

- 构建一个可长期维护的个人内容站
- 保持极简、稳定、低运维成本
- 以内容生产为中心，而非技术堆砌
- 让“更新流程”足够顺滑，降低发布阻力

### 1.2 非目标（明确不做）

- 不引入数据库或复杂后端
- 不做高耦合业务系统
- 不追求一次性大而全功能

### 1.3 约束条件

| 约束 | 含义 | 对实现的影响 |
|---|---|---|
| 最小侵入改造 | 尽量复用原主题结构 | 优先改配置与内容，少改核心布局 |
| 低运维 | 静态托管 + Git 发布 | 不增加服务端维护负担 |
| 快速可见 | 修改后尽快上线验证 | 采用小步提交、持续推送 |
| 内容优先 | 先确保写作与展示链路 | 功能扩展排在内容体系之后 |

## 二、信息架构设计（IA）

站点采用“按内容类型组织”的信息架构：

| 栏目 | 路由 | 角色 | 阅读意图 |
|---|---|---|---|
| 随笔 | `/essay/` | 长文主阵地 | 深度阅读 |
| 项目 | `/projects/` | 项目沉淀区 | 快速理解“做了什么、怎么做” |
| 絮语 | `/bits/` | 短内容流 | 轻量浏览 |
| 小记 | `/memo/` | 时间片段记录 | 回看与归档 |
| 归档 | `/archive/` | 索引层 | 按时间检索 |
| 关于 | `/about/` | 身份入口 | 了解站点与联系方式 |

**设计原则**：写作场景与阅读场景分离。作者按内容类型发布，读者按消费方式进入。

## 四、技术选型与取舍

### 4.1 为什么选择 Astro

- 内容站天然契合：Markdown、内容集合、静态输出都成熟
- 性能稳定：静态页面首屏快、托管简单
- 维护友好：目录清晰、改动成本可控

### 4.2 为什么采用主题二次开发

- 原主题已具备双栏布局、暗色模式、阅读模式、RSS 基建
- 二次开发可更快进入“内容生产阶段”
- 能把工程精力集中在“站点语义与内容质量”而非重复造轮子

### 4.3 为什么不引入后端

- 当前需求以内容发布为主，不需要动态数据写入
- 静态方案在可靠性、成本、维护复杂度上更优
- 保持架构轻量，便于长期个人维护

### 4.4 评论系统决策：Giscus

- 依托 GitHub Discussions，避免自建账号系统
- 审核、管理、迁移成本低
- 与项目仓库天然绑定，便于后续维护

| 字段 | 作用 |
|---|---|
| `title` / `description` / `date` | 基础展示与排序 |
| `status` | 项目状态管理（planned/in-progress/shipped...） |
| `stack` | 技术栈结构化展示 |
| `links.github` / `links.demo` | 项目入口与复现路径 |
| `tags` / `featured` / `slug` | 推荐、筛选与路由稳定性 |

**收益**：后续可以直接做过滤、置顶、关联，不依赖正文语义解析。

## 七、质量策略（可读性优化方法）

为避免“流水账式项目记录”，本页采用了统一写作规范：

1. **先结论后过程**：读者先拿到关键信息，再决定是否深入。
2. **阶段化表达**：Plan → 执行 → 验收，避免叙述跳跃。
3. **表格化信息**：约束、选型、风险用表格统一承载。
4. **语义一致性**：全站统一“小圈”术语，减少认知噪声。
5. **可复用沉淀**：把一次性动作抽象为可迁移方法。

## 八、风险清单与演进路线

### 8.1 当前风险

| 风险 | 触发条件 | 影响 | 应对 |
|---|---|---|---|
| 内容增长后导航拥挤 | 文章数量持续增加 | 查找效率下降 | 增加标签/筛选层 |
| 图片体积失控 | 无统一压缩规范 | 页面体积增大 | 建立图片压缩准入标准 |
| 更新节奏中断 | 单人维护时间波动 | 内容停更 | 采用“短内容优先”的轻量更新策略 |

### 8.2 下一阶段路线（Roadmap）

- 增加项目页内容密度（2~3 篇项目文档）
- 补充项目与随笔的关联链路
- 继续完善 RSS / canonical / sitemap 的一致性
- 逐步完善品牌视觉资产（favicon、封面规范）

## 九、可迁移经验（可复用到其他内容站）

1. 先做语义迁移，再做视觉精修。
2. 先删模板痕迹，再补真实内容，效率更高。
3. 用结构化 frontmatter 提前为未来功能预留空间。
4. 小步上线优于大批量离线改造，回归风险更低。
5. 个人站点的长期价值来自持续输出，而非技术复杂度。
