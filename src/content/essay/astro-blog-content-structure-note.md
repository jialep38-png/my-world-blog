## 为什么先整理结构

整理个人博客时，我越来越觉得：内容系统最重要的不是一开始就复杂，而是要有一层稳定结构。

对 Astro 这类以内容文件为核心的博客来说，Markdown 本身已经足够轻。真正容易混乱的地方，往往不是写文章，而是写完之后不知道它属于哪里、以后如何找到、页面如何展示。

所以这次我优先整理内容结构，而不是先改视觉或功能。

## 先固定内容类型

目前我把内容分成三类：

- `essay`：完整一点的文章，可以表达观点和过程；
- `projects`：围绕具体项目，记录目标、进展和下一步；
- `memo`：更短的记录，保留时间点和上下文。

这三类不是严格边界，而是写作入口。它们的作用是让每篇内容有一个默认位置。

当入口稳定以后，写作时就少了一层摩擦：不用先想“这个页面要怎么设计”，只需要判断“它属于哪一种记录”。

## frontmatter 是内容契约

Markdown 内容最容易被忽视的是 frontmatter。它看起来只是几行元数据，但会影响列表页、详情页、归档页和后续迁移。

我现在比较在意这几个字段：

```yaml
title: "文章标题"
description: "一句话摘要"
date: 2026-05-10
tags: ["博客"]
draft: false
archive: true
slug: "stable-url-slug"
```

这里最重要的是 `slug` 和 `date`。前者决定链接是否稳定，后者决定内容在时间线里的位置。标题以后可以微调，但链接最好不要频繁变化。

## 正文也需要一个小模式

一篇技术相关的个人记录，不一定要写得很长，但最好能回答几个固定问题：

1. 为什么要做？
2. 做了什么？
3. 哪些地方值得记下来？
4. 下一步是什么？

这个结构足够小，不会让写作变成负担，也能避免文章只剩零散结论。

## 不急着过度设计

个人博客很容易在“整理内容”时滑向“重做系统”。但对我来说，当前更重要的是让内容重新流动起来。

稳定结构的价值在于：以后无论是改样式、加索引、做标签页，还是迁移内容，都有比较清楚的基础。

先把文章放对位置，先让链接稳定，先让自己愿意继续写。技术系统服务于这个目标，而不是反过来变成新的阻力。
